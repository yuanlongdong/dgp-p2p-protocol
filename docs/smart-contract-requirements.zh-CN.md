# DGP-P2P 智能合约需求文档（仲裁 / 信誉 / 交易管理）

## 1. 文档目标与范围

本文档定义 DGP-P2P 交易场景下的核心智能合约需求，覆盖以下模块：

1. 仲裁模块（Arbitration Module）
2. 信誉评分系统（Reputation Scoring System）
3. 交易管理系统（Trade Management / Escrow）
4. 合约安全性与异常处理要求

目标是形成可直接用于合约设计、实现和测试验收的需求基线。

---

## 2. 角色定义

- **买家（Buyer）**：发起或参与交易的一方，支付资金并确认履约。
- **卖家（Seller）**：提供商品/服务并等待托管资金释放的一方。
- **仲裁参与者（Voter / Mediator）**：在争议场景中投票裁决。
- **协议管理员（Admin / Governance）**：仅用于参数配置（例如仲裁时限、投票阈值），不直接决定单笔争议输赢。

---

## 3. 总体业务流程

1. 创建交易：记录交易 ID、买卖双方地址、金额、托管状态。
2. 资金托管：买家向合约注资。
3. 正常完成：买家确认后释放资金给卖家；或按规则退款。
4. 争议触发：任一方在有效期内发起争议并提交证据。
5. 仲裁投票：合资格参与者投票（支持买家/卖家）。
6. 仲裁执行：根据投票结果与规则分配资金并记录结论。
7. 信誉更新：根据交易结果、纠纷记录、投票行为动态更新信誉分。

---

## 4. 功能模块需求

## 4.1 仲裁模块

### 4.1.1 功能描述

用于自动化解决买卖双方交易纠纷，输出透明、可验证、可追溯的仲裁结果。

### 4.1.2 输入

- `tradeId`：交易 ID（唯一）
- `votesForBuyer`：支持买家票数（或投票集合）
- `votesForSeller`：支持卖家票数（或投票集合）
- 可选扩展输入：证据哈希（如 IPFS CID）、投票截止时间、最小法定人数（quorum）

### 4.1.3 输出

- 仲裁结果（`winner = buyer | seller | tie`）
- 仲裁金额分配（例如：全额给买家、全额给卖家、按比例分配）
- 结案状态与时间戳
- 对应事件日志（便于链上审计与前端检索）

### 4.1.4 核心规则

1. **公平投票机制**
   - 一地址一票（或按明确权重规则），禁止重复投票。
   - 投票窗口固定：开始时间与截止时间在链上可验证。
   - 投票前后状态可查询，结果可复算。

2. **透明仲裁流程**
   - 发起争议、投票、结案均需触发事件。
   - 每个状态变更必须可追溯（谁、何时、对哪个 tradeId）。

3. **结果确定性**
   - 达到法定人数后，按规则判定胜诉方。
   - 平票处理策略必须预先定义（例如默认退款、进入二次仲裁、按比例分配）。

4. **时限处理**
   - 超时无人投票或票数不足时，按预设 fallback 策略自动结案。

### 4.1.5 事件建议

- `DisputeOpened(tradeId, buyer, seller, openedAt)`
- `VoteCast(tradeId, voter, side, weight)`
- `DisputeResolved(tradeId, winner, buyerAmount, sellerAmount, resolvedAt)`

### 4.1.6 验收标准

- 同一投票人不能对同一争议重复投票。
- 投票结束前不得执行结案。
- 争议结案后不能再次修改结果。
- 结案金额总和必须等于交易托管金额（考虑手续费时需显式列出公式）。

## 4.2 信誉评分系统

### 4.2.1 功能描述

基于交易历史、纠纷结论与反馈行为，为买卖双方提供动态信誉评分。

### 4.2.2 输入

- 买卖双方交易记录（完成次数、取消次数、超时次数）
- 争议记录（发起次数、胜诉次数、败诉次数）
- 投票记录（可选：作为仲裁参与者时的行为质量）

### 4.2.3 输出

- 用户信誉分（建议 0–1000 或 0–100）
- 最近更新时间戳
- 评分组成明细（便于解释和审计）

### 4.2.4 评分规则（建议）

可采用线性加权模型（示例）：

- 基础分：`base = 600`
- 成功完成交易奖励：`+a * completedTrades`
- 争议败诉惩罚：`-b * lostDisputes`
- 超时/违约惩罚：`-c * timeoutOrDefaultCount`
- 正向反馈奖励：`+d * positiveFeedback`

最终分数：

`score = clamp(minScore, maxScore, base + rewards - penalties)`

> 注：参数 `a,b,c,d,minScore,maxScore` 应可治理配置，并可通过事件记录参数变更历史。

### 4.2.5 自动更新触发点

- 交易成功释放资金时更新双方评分。
- 交易退款或超时关闭时更新评分。
- 仲裁结案时按胜诉/败诉结果更新评分。

### 4.2.6 验收标准

- 每次状态变更后，评分更新逻辑可复算、一致、可追溯。
- 评分上下限严格生效，不出现越界。
- 评分更新与交易/仲裁事件保持一致。

## 4.3 交易管理系统

### 4.3.1 功能描述

管理每笔交易的全生命周期，记录托管资金状态与交易状态变更。

### 4.3.2 输入

- `tradeId`
- `amount`
- `status`（待处理、已托管、已释放、已退款、争议中、已仲裁）
- 买卖双方地址

### 4.3.3 输出

- 交易状态更新结果
- 托管资金状态（锁定金额、可释放金额、已释放金额）
- 事件日志与可查询记录

### 4.3.4 状态机定义（建议）

- `Pending`：交易创建但未注资
- `Funded`：已注资托管
- `Released`：资金已释放给卖家（终态）
- `Refunded`：资金已退回买家（终态）
- `Disputed`：争议处理中
- `Arbitrated`：仲裁已结案（终态，且资金已按裁决分配）

### 4.3.5 状态迁移约束

- `Pending -> Funded`
- `Funded -> Released`
- `Funded -> Refunded`
- `Funded -> Disputed -> Arbitrated`

禁止：

- 终态后再次迁移（如 `Released -> Disputed`）
- 在未注资状态下释放或退款

### 4.3.6 验收标准

- 任一 `tradeId` 只能有唯一有效交易实体。
- 合约余额与所有托管资金汇总一致。
- 任意非法状态迁移必须回滚。

## 5. 安全性要求

### 5.1 重入攻击防护

- 对资金转账函数使用重入锁（如 `nonReentrant`）。
- 遵循 Checks-Effects-Interactions 模式：先校验、再更新状态、后外部调用。
- 尽可能使用“拉取式提现（pull over push）”降低外部调用风险。

### 5.2 溢出/下溢防护

- 使用 Solidity 0.8+ 默认算术检查。
- 对关键计算（分账、权重、费率）增加边界校验。
- 对比例计算采用统一精度（如 bps）并防止精度丢失导致金额不守恒。

### 5.3 访问控制与权限最小化

- 管理员仅可修改系统参数，不可篡改单笔交易结果。
- 关键配置变更需事件记录，建议配合多签与 timelock。

### 5.4 异常处理

- 对无效输入（不存在 `tradeId`、非法地址、金额为 0、重复操作）必须 `revert`。
- 对超时、票数不足、平票等异常路径应有明确 fallback 规则。
- 避免静默失败，错误信息应便于定位。

### 5.5 可观测性与审计

- 所有关键动作必须发事件：创建、注资、释放、退款、争议、投票、结案、评分更新。
- 事件字段需包含 `tradeId` 与参与方地址，确保离线索引与审计可用。

---

## 6. 非功能性要求

- **Gas 成本可控**：投票、结案、评分更新在可接受成本范围内。
- **可升级策略清晰**：若采用代理模式，需定义升级权限与流程。
- **兼容性**：支持主流 EVM 链与区块浏览器事件检索。
- **可测试性**：单元测试需覆盖主流程、异常流程与恶意路径。

---

## 7. 测试与验收用例（最小集）

1. 正常交易：创建 -> 注资 -> 释放 -> 双方评分更新。
2. 退款流程：创建 -> 注资 -> 退款 -> 状态闭环正确。
3. 争议流程：创建 -> 注资 -> 争议 -> 投票 -> 结案 -> 分账正确。
4. 重复投票防护：同地址重复投票应回滚。
5. 非法状态迁移：终态后任意操作应回滚。
6. 超时 fallback：票数不足或超时后按规则自动处理。
7. 安全测试：重入攻击模拟、异常输入、边界金额与极值参数。

---

## 8. 建议的里程碑交付

- **M1（需求冻结）**：确认状态机、仲裁规则、评分公式。
- **M2（合约实现）**：完成核心合约与事件定义。
- **M3（测试与审计准备）**：完成测试矩阵、安全基线检查。
- **M4（上线准备）**：参数治理、运维 runbook、监控告警联调。
